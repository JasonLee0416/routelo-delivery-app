# OCR repository dataset experiment — 2026-06-23

## 결론

현재 Android ML Kit 연결은 실제 저장소 이미지 8장 모두에서 텍스트를 반환했다. 앱에 있던 데모 주문번호, 전화번호, 주소, 예식장명이 결과에 섞인 사례는 0건이었다.

그러나 구조화 파서는 아직 운영 투입 가능한 상태가 아니다. 이미지에 존재하는 문자열을 잘못된 필드로 배치하는 명확한 거짓 데이터 위험이 10건, 수신자 정보라고 단정할 수 없는 전화번호 대입이 2건 발견됐다. 즉, 하드코딩된 샘플을 생성하는 문제는 재현되지 않았지만, 실제 OCR 문자열을 잘못 해석해 사실처럼 표시하는 문제는 재현됐다.

## 실험 전 상태

- 기준 커밋: `b7b63a9992a4e5c3fec477b53ed0ffdec10c3ca3`
- 기준 브랜치: `codex/critical-device-fixes`
- Android 네이티브 모듈에 `com.google.mlkit:text-recognition-korean:16.0.1` 연결 완료
- 앱의 이미지 URI를 ML Kit에 전달하고 원문, 줄, 좌표를 TypeScript 파서에 전달하는 경로 구현 완료
- 단위 테스트에서는 명시적으로 넣은 데모 문자열과 recognizer mock만 검증
- 저장소의 실제 시험 이미지에 대한 네이티브 실행 증거는 0건
- 필드 단위 정답표(ground truth)는 아직 없음

따라서 이번 실험은 먼저 다음 두 가지를 확인하는 기준선 실험으로 정의했다.

1. 실제 이미지에서 네이티브 OCR이 빈 결과 없이 동작하는가?
2. 결과에 앱 데모값 또는 원문으로 뒷받침할 수 없는 값이 들어가는가?

정답표가 없으므로 문자 인식률(CER)과 필드 정확도(F1)는 이번 결과에서 계산하지 않았다.

## 실행 과정

1. Android API 35 x86_64 에뮬레이터를 headless 모드로 부팅했다.
2. 저장소 이미지 8장을 빌드 시점에 Android instrumentation test asset으로 복사했다.
3. 각 이미지를 `BitmapFactory`로 디코딩한 뒤 동일한 한국어 ML Kit recognizer에 순차 전달했다.
4. 이미지 SHA-256, 크기, 처리 시간, 전체 텍스트, 줄 텍스트와 bounding box를 JSON으로 저장했다.
5. 앱의 현재 `parseReceiptText` 함수에 각 전체 텍스트를 그대로 전달했다.
6. 각 구조화 필드의 값, 신뢰도, sourceText, 원문 추적 가능 여부를 기록했다.
7. 데모 fixture의 고유 문자열 4개가 원문 또는 구조화 결과에 들어갔는지 검사했다.
8. 전화번호 형식, 주문번호의 전화번호 오인, 레이아웃 이동에 의한 수령인 오인 등 명확한 의미 오류를 별도로 집계했다.

첫 실행은 `/data/local/tmp` 파일을 instrumentation 프로세스에서 읽는 단계에서 디코딩 실패했다. OCR 실패가 아니라 Android 테스트 프로세스의 파일 접근 경계 문제였다. 입력 파일을 수정하지 않고 빌드 시 테스트 APK asset으로 복사하는 방식으로 변경한 뒤 같은 조건으로 재실행했고 성공했다.

## 발견된 문제

### 1. 주문번호가 수령인 전화번호로 변환됨

`_02`, `_03`, `_05`, `_06`에서 주문번호의 `202606...` 일부가 `02-606-2606`으로 정규화됐다. 전화번호 정규식에 숫자 경계 조건이 없고, 라벨 없는 첫 전화 후보를 수령인 전화번호로 대입하기 때문이다.

### 2. 열 기반 양식이 행 기반 파서에서 밀림

현재 파서는 `라벨` 다음 줄을 그 값으로 해석한다. 시험 인수증은 왼쪽에 라벨이 연속되고 오른쪽에 값이 모이는 열 기반 양식이 많다. 이 때문에 다음과 같은 오분류가 발생했다.

- `99플라워` → 수령인 이름
- `(분시 이름)`, `(반드시 이)` 같은 양식 안내 → 수령인 이름
- 발주처/연락처 줄 → 메모

### 3. 타입 검증 전에 높은 신뢰도가 부여됨

`_07`에서는 날짜와 시간 문장이 수령인 전화번호 필드에 들어갔는데도 신뢰도 96이 부여됐다. 필드 라벨 매칭 점수와 값 타입 유효성 검사가 분리되지 않았기 때문이다.

### 4. 업무 스키마와 현재 파서 필드가 맞지 않음

시험 인수증의 핵심 라벨인 발주화원, 배송화원, 상품명, 수량, 배달일시, 리본 문구를 현재 9개 필드 스키마가 충분히 표현하지 못한다. 원문 인식은 됐지만 상당수가 `unmapped`로 남았다.

## 판정

- 네이티브 ML Kit 연결: **동작**
- 저장소 8장에 대한 텍스트 인식: **8/8 동작**
- 과거 데모 데이터의 자동 삽입: **재현되지 않음**
- 원문에 없는 완전히 새로운 문자열 생성: **재현되지 않음**
- 잘못된 필드 분류를 사실처럼 표시하는 문제: **재현됨**
- 자동 저장 및 자동 배달 등록에 사용 가능한 수준: **아직 아님**

현재 결과는 “OCR 엔진이 거짓말한다”기보다 “파서가 실제 문자열의 의미를 잘못 붙여 거짓 정보처럼 만든다”에 가깝다. 사용자에게는 두 경우 모두 잘못된 데이터로 보이므로 자동 등록 전에 차단해야 한다.

## 권장 조치

1. 전화번호 정규식에 숫자 경계를 추가하고 주문번호 줄에서 발견된 후보를 제외한다.
2. 전화번호, 날짜, 시간 등 타입 검증을 통과하지 못하면 값을 비우고 신뢰도를 0으로 만든다.
3. ML Kit의 bounding box를 사용해 같은 행의 라벨과 값을 연결하는 2D 레이아웃 파서를 만든다.
4. 발주화원/전화, 배송화원/전화, 상품명, 수량, 배달일시, 주소, 요청사항, 인수자 전화 필드를 스키마에 추가한다.
5. 필수 필드가 낮은 신뢰도이거나 출처 라벨이 없으면 자동 저장하지 않고 사용자 검토 화면을 표시한다.
6. 8장에 대한 필드별 정답 JSON을 작성한 뒤 CER, field precision/recall/F1을 계산한다.
7. 수정 후 동일 SHA-256 데이터셋과 동일 에뮬레이터에서 다시 실행해 이번 기준선과 비교한다.
